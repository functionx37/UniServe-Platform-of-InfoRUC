# 后续同学接手说明

这份说明是给后续开发同学直接接手用的。你们如果要继续做管理后台，不需要重新理解整套页面，只需要按下面顺序阅读即可。

## 一、先看哪些文件

### 1. 必看文件

1. `管理后台接口对齐说明.md`
   - 这是接口标准的主文档。
   - 后端字段名、请求体、返回体都以这个文件为准。

2. `admin-frontend/src/api/adminApi.ts`
   - 这是前端唯一的接口入口。
   - 当前是 Mock 实现。
   - 后续如果接真实后端，只改这个文件即可，页面层不要重写。

3. `admin-frontend/src/App.tsx`
   - 用来确认页面到底消费了哪些数据。
   - 如果不改页面逻辑，可以只大致浏览。

4. `postman/管理后台接口.postman_collection.json`
   - 用来联调和测试接口。
   - 后端起服务后，直接改 `baseUrl` 和 `token` 即可。

### 2. 选看文件

1. `README-同学D交付说明.md`
   - 看整个交付目录和运行方式。

2. `qa/压力测试报告.md`
   - 看压测结果该怎么填。

3. `qa/k6-精准推送压力测试.js`
   - 后端准备好后可以直接跑压测。

## 二、谁该看什么

### 后端同学

重点看：

- `管理后台接口对齐说明.md`
- `postman/管理后台接口.postman_collection.json`

你们要做的事：

- 按文档把接口实现出来
- 统一返回 `success/message/data`
- 保证字段名完全一致
- 完成权限校验和操作日志记录

### 前端同学

重点看：

- `admin-frontend/src/api/adminApi.ts`
- `admin-frontend/src/App.tsx`

你们要做的事：

- 把 `adminApi.ts` 里的 Mock 逻辑替换成真实请求
- 页面层尽量不要改字段名
- 如果接口字段有变动，先回到接口说明文档统一修改，不要私自改页面结构

### 测试同学

重点看：

- `postman/管理后台接口.postman_collection.json`
- `qa/k6-精准推送压力测试.js`
- `qa/压力测试报告.md`

你们要做的事：

- 用 Postman 做功能联调
- 用 `k6` 跑并发测试
- 把结果填进测试报告模板

## 三、真正需要改的文件

如果后端开始开发，前端侧原则上只改下面这个文件：

- `admin-frontend/src/api/adminApi.ts`

原因：

- 页面已经按统一数据结构接好了
- 导入、推送、日志、仪表盘都通过这个文件拿数据
- 所以后续不要去 `App.tsx` 里找接口调用分散着改

## 四、推荐接手流程

1. 先读 `管理后台接口对齐说明.md`
2. 再看 `admin-frontend/src/api/adminApi.ts`
3. 后端照着实现接口
4. 前端把 `adminApi.ts` 改成真实请求
5. 用 Postman 跑一遍
6. 用 `k6` 跑压力测试

## 五、当前已定下来的规则

为了避免后面越改越乱，这几个规则默认不要改：

- 所有管理端接口统一走 `/admin/*`
- 所有响应统一走 `success/message/data`
- 推送筛选条件统一用 `grade/major/identity`
- 通知列表统一字段为
  - `id`
  - `title`
  - `category`
  - `grade`
  - `major`
  - `channel`
  - `publishAt`
  - `status`
- 发送日志统一字段为
  - `id`
  - `title`
  - `audience`
  - `channels`
  - `sentAt`
  - `count`
  - `status`

## 六、你们如果时间不够，最少要完成什么

最低可联调版本只要完成这些接口：

1. `POST /auth/login`
2. `POST /admin/dashboard`
3. `GET /admin/notifications`
4. `POST /admin/import/notifications`
5. `GET /admin/import/sessions`
6. `POST /admin/push/preview`
7. `POST /admin/push/send`
8. `GET /admin/push/logs`

这样前端页面就能基本完整跑通。
