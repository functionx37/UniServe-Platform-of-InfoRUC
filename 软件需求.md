1. 引言
1.1 编写目的
本文档为“中国人民大学信息学院学生综合服务与党团管理平台"的软件需求规格说明书。编写本文档的主要目的在于将前期产品需求文档及团委老师针对“微人大”系统对接限制、四级权限划分、人大金仓数据库适配等提出的具体补充要求，转化为标准化的技术规格描述。
具体目标包括:
1.界定技术实现边界:明确信息学院内部事务处理(如分级审批流程、PDF成绩单解析、精准标签推送等)的功能范围，为后续的系统设计与编码提供排他性的技术准则。
2.指导多端协同开发:为前端(微信小程序)、后端(PC 后台管理系统)以及 AI 算法模块的参与人员提供统一的开发蓝图，确保系统在国产化数据库环境下的稳定运行与数据一致性。
3.确立安全性与隐私规范:针对人大信息学院学生的身份证号、手机号、户籍及处分信息等敏感数据，制定明确的加密存储与权限脱敏规则，确保系统设计符合学校及学院的数据安全管理规定。
4.支撑课程大作业验收:作为软件工程课程末期项目评审、答辩及甲方(学院分团委)验收的正式依据。本文档将作为评估系统“实用性"与“完整度"核心指标的对照基准。

1.2 读者对象
本说明书面向该平台开发与评审全过程中的各类相关参与人员，具体读对象及用途如下:
1.系统研发参与人员:包含负责前端(微信小程序)、后端(PC 后台)、数据库及 AI算法与问答模块的开发成员。本文档为其提供业务逻辑细节及四级权限管控的实现基准，确保各端协同开发。
2.质量保障与测试人员:作为编写集成测试用例与安全性评估报告的唯一权威准则。通过对照本文档定义的各项功能指标，验证系统对于身份证号等敏感数据的加密逻辑、分级审批流的闭环性以及高并发下的系统稳定性。
3.课程教学与评审团队:包含信息学院软件工程课程的指导教师、助教及最终答辩评审专家。本文档是其监控项目进度、审核技术方案合规性，以及对交付物“实用性"与“完整度"两项核心指标进行评分的官方审计依据。
4.业务需求审定方(学院分团委、辅导员及助教):作为学院学生事务实际管理者，其通过阅读本文档核实系统设计是否精准覆盖了入党流程追踪、学分缺口预警及"一站式"证明生成等实际业务场景，确保最终产品能真正解决办公痛点。
1.3 软件项目概述
项目名称:中国人民大学信息学院学生综合服务与党团管理平台
用户单位:中国人民大学信息学院(涵盖全院约1200名本科生及研究生，以及分团委老师、班主任、相关学生工作助教等)
开发单位:萌萌软工少女团！
大致功能和用途:
本系统旨在打造信息学院“一站式"事务处理数字化窗口，作为校级"微人大"系统的补充与延伸解决学院本地学生事务中存在的流程不透明、信息滞后及重复性劳动多等痛点。核心用途包括:
1.智能政策咨询中心:利用 AI匹配技术构建学院政策库，针对奖助学金申请、休学复学、宿舍调整等高频咨询提供即时、标准化的回复。
2.党团建设全生命周期管理:实现入党、入团全流程的可视化追踪(从入党申请人到正式党员)，并根据学院管理节点自动触发提醒。
3.精准化信息触达体系:利用标签化分发技术，将学院通知、实习就业、学术竞赛等信息，根据年级和专业维度精准推送至目标学生群体。
4.电子化审批与证明生成:数字化学院内部审批流(如请假、盖章、入党时间证明等)，系统基于 Kingbase 数据库自动填充标准化模板并生成 PDF 文档。
5.学业分析与选课预警:针对信息学院各专业培养方案，通过解析 PDF 成绩单，自动进行学分比对并亮显缺口模块，辅助学生进行精准选课规划。
1.4 文档概述
本文档系统化地规范了平台的技术实现路径与业务逻辑，确保系统开发与学院实际办公需求高度对齐。文档整体组织结构如下:
第一部分:引言。阐述文档的编写背景、目的及参与人员，并对系统中涉及的专门术语进行规范性定义。
第二部分:软件的一般性描述。界定系统与校级"微人大"系统的外部接口关系(基于文档导入导出)，说明运行环境(微信小程序+PC 端)及项目实施过程中的主要约束与限制条件。
第三部分:软件功能需求描述。本文档的核心章节。通过用例建模与交互分析，详细规定智能问答库、党团流程可视化、信息精准分发、电子审批流以及学业预警五大核心模块的具体业务规程。
第四部分:其它软件需求描述。明确系统的非功能性指标，重点涵盖安全性(身份证等敏感数据加密)、性能要求、界面设计要求(遵循人大严谨正式风格)以及期末交付物的质量标准。
第五部分:软件原型。提供系统关键操作路径的原型图示，直观展现从学生提交申请至老师多级审批完成的交互流程。

1.5 定义
[1]高并发处理能力：指系统在奖助学金评定、党团周报提交等业务高峰期，需具备支撑约 200 人同时访问、向上兼容 1200 名师生并发请求的稳定性，确保系统不崩溃、不宕机。
[2]模块化开发：指将系统划分为智能问答、党团管理、信息推送等独立的功能单元。该方式旨在确保系统具有良好的可维护性和扩展性 (Maintainability and Extensibility)，以便后续快速迭代如“出国申请”等新功能。
[3]死锁：指在金仓数据库（Kingbase）环境下，因高频 IO 操作或多端并发导致两个及以上事务互相等待资源而停滞的状态。系统需通过优化代码结构予以规避，确保数据状态实时同步。
[4]Vue.js / React：系统 PC 端（管理员端）可选用的主流前端框架，用于构建负责处理复杂统计报表、文档导入导出的单页面应用。
[5]Spring Boot / FastAPI：系统后端（服务端）可选用的模块化架构。前者适用于复杂的业务逻辑分发，后者适用于高性能的 API 接口响应与 AI 模块集成。
[6]脱敏处理：指针对身份证号、联系方式、处分记录等敏感信息，在前端展示时通过部分遮蔽（如“1101...001X”）或动态隐藏进行的保密技术，防止隐私泄露。
[7]“补丁式”开发：指缺乏整体规划、仅针对局部逻辑漏洞进行临时修补的低质量编码行为。本系统严禁此类开发，要求代码具备高可读性与完善的文档注释。
[8]呼吸灯效果：用于党团流程可视化界面的 UI 交互效果。指通过光晕亮灭的节奏变化，清晰标注学生当前所处的阶段（如“积极分子”阶段），增强引导感。
[9]Webview：指小程序内用于承载 H5 网页的组件。系统通过该组件实现响应式适配，确保学生在移动端能流畅填写请假申请或下载模板。
[10]压力测试：指在系统集成阶段，通过模拟高强度负载来验证四级权限下系统响应逻辑的鲁棒性，并修复 1、2 级关键 Bug。
[11]SQL 文件：指交付物中包含的数据库初始化脚本。用于在 Kingbase 环境下快速还原表结构及预置的党政知识库数据。
[12]Docker 镜像：系统交付的部署形态之一。通过容器化技术，确保系统在学院 Linux 服务器环境下能跨平台、零配置直接运行。
[13]WAR / Jar 包：Java 应用程序的归档格式。作为系统后端逻辑的最终交付物，部署于应用服务器（如 Tomcat）或直接通过 JVM 启动。
[14]交互协议：在《API 接口文档》中详细定义的规范。指前端（小程序/PC端）与后端（Server）、以及后端与 AI 模块之间进行数据交换的标准（如 RESTful 风格的 JSON 格式）。

2. 软件的一般性描述
2.1软件产品与其环境之间的关系
本软件为“学院学生综合服务与党团管理平台”，主要包含面向学生的用户端和面向老师/助教的管理员端。其与外部环境的关系如下：
外部用户：
终端用户（学生）：约1200名本科生及研究生，主要通过移动端（微信小程序）访问系统，进行事务办理、政策查询、进度追踪和学业分析。
管理员（老师/助教）：团委老师、班主任及相关助教，主要通过PC端浏览器访问后台管理系统，进行统计分析、文档导入导出、信息推送和审批操作。
硬件系统：
终端用户需使用支持微信运行的智能手机等移动设备。
管理员需使用连接互联网的个人电脑（PC）。
系统需部署在提供计算和存储资源的服务器上。
其他软件系统：
微信平台：用户端依托微信小程序运行，并基于微信账号进行实名制身份验证和消息推送。
数据库系统：底层依托人大金仓（Kingbase）数据库进行数据存储，保证数据的一致性和可靠性。
外部数据源与接口：由于无法直接对接校级“微人大”系统接口，系统需支持通过Excel、Word、PDF等格式的文档与外部系统进行离线的数据交互（导出与批量导入）。
2.2限制与约束
接口与数据限制：因无法直接调用校级“微人大”系统接口，数据的同步与流转必须依赖手工或批量导入/导出（Excel/Word/PDF），这限制了数据的实时互通性。
安全性与隐私约束：涉及学生隐私的基本信息（如身份证号、联系方式等）必须严格加密存储，保障数据安全。
大模型应用限制：在智能咨询与问答模块中，回复原则必须优先基于标准答案或官方文件链接，严格避免大模型产生“幻觉”；敏感信息不得直接展示，仅提供官方链接。
并发与性能约束：系统预计日均同时在线人数较少（约几十人），因此对高并发处理能力[1]要求不高，重点在于系统的稳定性和业务逻辑的准确性。
开发与架构限制：要求采用模块化开发[2]，必须保证系统具有良好的可维护性和扩展性，以便后续迭代增加新功能（如出国申请、评价评优等）。
2.3假设与前提条件
用户基础假设：假设所有目标终端用户（学生）均拥有微信账号，并愿意授权小程序进行实名制登录和身份验证。
管理端条件：假设管理员具备基本的信息化办公能力，能够熟练操作PC端后台进行数据的审批、标签化分发及文档的导入导出。
运行环境前提：假设系统部署所需的服务器环境及人大金仓（Kingbase）数据库已准备就绪并能稳定运行。
数据来源假设：假设学院能够提供完整、准确的政策文件、常用模板、官方题库及各年级专业的培养方案，以供系统录入和知识库构建。
外部文件格式假设：假设学生上传的成绩单等文件符合系统支持的PDF标准格式，以便系统能够正确解析并进行学业情况分析。

3. 软件功能需求描述
3.1 软件功能概述
3.1.1 功能总体说明
本系统为“学院学生综合服务与党团管理平台”，面向信息学院一般学生与老师等管理员两类用户，提供集事务办理、信息服务、数据分析与流程管理于一体的一站式服务。系统主要包括以下五大功能模块：
1.智能问答与政策知识库
2.党团事务流程管理
3.信息集成与精准推送
4.电子证明生成与审批流程
5.学业情况分析与预警
各功能模块相互协同，共同提升学院学生事务管理的信息化与自动化水平。
3.1.2 功能列表
①智能问答与政策知识库模块
F-01 智能政策问答
功能描述：系统基于关键词匹配与知识库检索机制，为用户提供政策咨询服务。优先返回标准答案或官方文件链接，避免不确定性回答。
优先级：高
使用场景：学生查询奖助学金政策、放假安排、学籍变动流程等信息。
F-02 政策知识库管理
功能描述：管理员可上传、更新、删除政策文件（支持Word、PDF等格式），构建可查询的知识库。
优先级：高
使用场景：管理员维护学院政策文件及通知资料。
F-03 模板文件下载
功能描述：提供常用文档模板下载（如证明模板、请假条、活动预算表等）。
优先级：中
使用场景：学生下载规范格式文档用于事务办理。
②党团事务流程管理模块
F-04 流程可视化展示
功能描述：系统展示入党、入团等流程的标准路径及阶段划分。
优先级：高
使用场景：学生了解发展流程整体结构。
F-05 个人进度查询
功能描述：学生可查看自身当前所处流程阶段及已完成事项。
优先级：高
使用场景：学生跟踪入党或入团进度。
F-06 关键节点提醒
功能描述：系统基于时间规则自动触发提醒（如提交思想汇报等）。
优先级：中
使用场景：防止学生遗漏关键时间节点。
F-07 理论知识自测
功能描述：提供党建相关题库，支持学生在线测试与练习。
优先级：低
使用场景：学生进行理论学习巩固。
③信息集成与精准推送模块
F-08 信息采集与录入
功能描述：管理员录入或导入各类通知信息（如校园通知等）。
优先级：高
使用场景：汇总多来源通知信息。
F-09 信息标签分类
功能描述：对信息进行标签化处理（如“就业”、“实习”、“计算机类”等）。
优先级：高
使用场景：实现后续精准推送。
F-10 精准信息推送
功能描述：根据学生画像（年级、专业等）定向推送信息。
优先级：高
使用场景：向特定群体发布通知。
F-11 多渠道通知发送
功能描述：支持通过系统消息、邮件等方式发送通知。
优先级：中
使用场景：提高信息触达率。
④电子证明生成与审批流程模块
F-12 电子证明自动生成
功能描述：系统自动填充学生信息生成标准证明文件，并支持PDF预览与导出。
优先级：高
使用场景：学生申请在读证明、政治面貌证明等。
F-13 审批流程管理
功能描述：支持电子化审批流程，管理员在线审核申请。
优先级：高
使用场景：教师审批学生提交的申请。
F-14 数据导入与导出
功能描述：支持Excel、Word等格式数据的批量导入与导出。
优先级：高
使用场景：管理员进行数据维护与统计。
⑤学业情况分析与预警模块
F-15 成绩数据上传与解析
功能描述：学生上传成绩单（PDF或Excel），系统自动解析数据。
优先级：高
使用场景：学生提交学业数据进行分析。
F-16 培养方案比对分析
功能描述：系统将成绩与培养方案进行对比，识别未完成模块。
优先级：高
使用场景：学生了解学分完成情况。
F-17 学业预警提示
功能描述：对未达标或存在风险的学业情况进行提示。
优先级：中

4. 其它软件需求描述
4.1 性能要求
4.1.1 并发支撑
预计日均同时在线人数较少（约几十人）；向上兼容最高量需支持约1200名师生在奖学金评定、党团周报提交等高峰期时并发访问，预计高峰期约两百人左右。
4.1.2 响应效率
普通页面跳转及数据查询响应时间应控制在1.5秒以内；智能问答（AI匹配）的首次响应时间不超过3秒；大规模 Excel/PDF 数据导入导出的处理时间，在网络正常情况下应在10秒内完成。
4.1.3 资源占用
服务器端需优化代码结构，确保在金仓数据库（Kingbase）环境下，高频 IO 操作不会导致数据库死锁[3]或内存溢出。
4.2 设计约束
4.2.1 开发工具与技术栈约束
多端协同架构：系统必须采用前后端分离架构。
①前端（移动端）：采用微信小程序框架，作为学生用户的高频操作入口，需利用微信实名制接口进行身份校验。
②前端（PC端）：采用主流 Web 开发框架（如 Vue.js 或 React[4]），作为管理员（老师/助教/班干）处理复杂业务、统计报表及文档导入导出的专用平台。
③后端（Server）：采用模块化后端架构（如 Spring Boot 或 FastAPI[5]），负责统一的业务逻辑处理与 API 分发。
AI 模块集成：AI 算法模块需通过标准化接口与后端对接，确保政策问答引擎能够高效读取知识库内容并过滤敏感信息。
4.2.2 运行环境约束
数据库环境：系统必须适配人大金仓（Kingbase）数据库。所有数据库操作需兼容国产数据库语法规范，确保在学校服务器环境下的稳定运行。
服务端环境：需支持在学校或学院提供的服务器（通常为 Linux 操作系统）上部署，需考虑与 Kingbase 数据库服务器的内网对接。
移动端：需在微信内置浏览器及主流手机浏览器（Safari, Chrome）中流畅运行。
PC 端：管理员后台需兼容 Chrome、Edge 及国产主流浏览器。
4.2.3 安全性要求（核心约束）
数据脱敏与加密：身份证号、联系方式、生源地、处分记录等敏感信息必须加密存储。在前端展示时，非管理员权限需进行脱敏处理[6]（如：1101...001X）。
身份认证：系统需实现基于学号/微信的一对一实名认证。
分级权限控制：系统必须严格执行 4 级权限校验逻辑（学院领导、管理老师、班团骨干、普通学生）。严禁助教及三/四级角色拥有审批权限，且敏感信息的展示需根据角色进行动态脱敏。
审计追踪：系统必须记录所有管理员（老师、助教、班干）的操作日志，确保审批操作和数据修改可追溯、可审计。
4.2.4 可靠性与维护性要求
代码稳健性：遵循软件工程规范，严禁“补丁式”开发[7]。代码需具备高可读性与完善的文档注释，确保系统的可持续维护性。
数据一致性：在多端并发（小程序、PC 端）环境下，需通过事务管理确保金仓数据库内的数据状态实时同步，避免产生流程死锁。
异常容错：针对误操作风险，审批流需具备 1-2 天内 的撤销或重批机制；在批量导入文档失败时，系统需提供精准的错误定位反馈。
4.3 界面要求
4.3.1 总体风格
视觉定调：系统界面应遵循“严谨、正式、简约”的设计原则。考虑到本平台不仅面向学生，也面向学院领导与老师，界面应避免过度活泼或低幼化的设计。
PC端：采用经典的“左侧导航+顶部状态+右侧内容”的后台管理布局，方便管理员进行大批量数据处理和统计分析。
移动端：采用“底部标签导航+顶部搜索区”的小程序式布局，确保单手操作的便捷性。
4.3.2 核心模块界面原型描述
智能问答界面：
采用类似对话框的交互方式，顶部提供高频问题快捷标签，底部对话框支持关键词智能关联提示。
党团流程可视化：
采用线性时间轴组件，清晰标注“入党申请人”、“积极分子”等 5 个阶段。已完成阶段使用颜色标注，当前阶段以呼吸灯效果[8]或图标标注，并显示预计完成时间。
学生画像展示：
综合运用雷达图（展示科研、竞赛、实践、成绩多维度）和数据列表。针对管理员可见的敏感信息（如生源地），设计专门的“查看”按钮及操作二次确认弹窗。
学业分析看板：
使用红/绿对比色高亮显示培养方案中“已修满/未修满”的学分模块。
4.3.3 交互约束
响应式适配：所有功能必须适配移动端 Webview[9]，确保学生在手机上填写请假申请或下载模板时，布局不乱、文字清晰。
易用性要求：核心业务逻辑（如政策咨询、证明生成）的点击深度不应超过 3次。
容错提示：在执行“导出敏感数据”或“撤回审批”等高风险操作时，必须弹出醒目的二次确认对话框。
4.4 进度要求
本项目的开发周期约为 16周（一学期），具体节点安排如下：
第 1-6 周（需求分析与原型设计）：完成详细需求规格说明书，产出 4 级角色的UI原型图，并完成数据库建模。
第 6-8 周（核心模块开发 - P0）：完成“智能问答知识库”及“党团事务线性流程管理”功能，实现管理员后台的数据批量导入。
第 9-12 周（次要模块与功能完善 - P1/P2）：开发奖励荣誉展示、学生画像管理；初步尝试学业预警功能；完成邮件通知接口开发。
第 13-14 周（系统集成与测试）：进行金仓数据库的兼容性测试，完成分级权限的压力测试[10]，修复 1、2 级 Bug。
第 15-16 周（交付与答辩）：撰写用户手册，准备演示 PPT，完成期末项目验收及甲方（团委老师）评审。
